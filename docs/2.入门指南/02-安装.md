# 安装

本文档用于说明 Mosure 的安装、初始化配置与部署要点。

你可以把安装过程理解成三件事：

- **把依赖装齐**（PHP / Composer；如需本地重新构建前端再使用 Node.js）
- **把运行环境配好**（主要是 `.env` 与数据库连接）
- **跑一次初始化**（通过命令行安装或页面安装，创建核心表、管理员账号，并生成 `.locked`）

如果你只是本地体验/开发，建议优先选择 SQLite，流程更轻量；生产环境更推荐 MySQL。

安装有两种方式，建议按场景选择：

- **方式 A：命令行安装**：不需要先启动 Web 服务，直接在项目目录执行命令即可完成初始化。
- **方式 B：页面安装**：需要先把项目跑起来（本地启动或完成服务器部署），再访问 `/install` 执行安装向导。


## 获取代码
 
 以仓库代码方式安装时：
 
```bash
git clone https://gitee.com/gibson_0822/mosure.git
cd mosure
```
 
 如果你是通过压缩包下载源码，请确保解压后目录中包含 `artisan`、`composer.json`、`package.json` 等文件。

## 安装方式 A：命令行安装（无需先启动项目）

本地体验推荐直接使用一键安装脚本，默认使用 SQLite，不依赖 Redis，也不需要先启动 Nginx/Apache。

```bash
# Linux / macOS
./bin/install.sh

# Windows
bin\install.bat
```

等价的 Artisan 命令：

```bash
php artisan mosure:install --name=Admin --email=admin@example.com --password=<your-password>
```

默认行为：

- 默认数据库为 SQLite；若未显式配置 `DB_DATABASE`，系统会自动使用项目目录下的绝对路径 `database/mosure.db`
- 默认 `QUEUE_CONNECTION=database`
- 默认 `CACHE_STORE=file`
- 本地运行使用 `php artisan serve`，无需额外配置 Nginx
- 仅当你需要重新构建前端资源且仓库中没有 `public/build` 时，才需要 Node.js 与 npm
- 安装器会主动创建 SQLite 文件并检查目录/文件是否可写
- 安装器会自动创建 `storage/framework/*`、`storage/logs`、`bootstrap/cache`

## 安装方式 B：页面安装（需先启动或部署）

页面安装依赖 HTTP 访问，因此你需要先让项目可访问，然后在浏览器打开安装向导 `/install`。

> 你可以先使用下文的“安装完成后：启动与部署方式”中的本地启动或部署方式把服务跑起来，再执行页面安装。

## 安装依赖
 
 ### 安装后端依赖
 
 后端依赖由 Composer 管理。首次安装会拉取 `vendor/` 下的依赖包。
 
 ```bash
 composer install
 ```

> ⏱️ **预计耗时**：首次安装可能需要几分钟，取决于网络速度和依赖包数量。如果遇到网络问题，可以考虑使用国内镜像源。


### 安装前端依赖
 
 前端依赖由 npm 管理。只有在你需要重新构建前端资源时才需要安装。
 
```bash
npm install
```

> ⏱️ **预计耗时**：首次安装通常需要 1-3 分钟。如果速度较慢，可以考虑使用淘宝镜像源（`npm config set registry https://registry.npmmirror.com`）。

## 运行环境配置（.env）
 
 `.env` 是 Laravel 项目最重要的运行环境文件：数据库连接、应用 URL、缓存和队列驱动等都在这里决定。
 
 - 示例文件：`.env.example`
 - 默认行为：首次安装依赖时，Composer 脚本会在 `.env` 不存在时从 `.env.example` 复制生成 `.env`
 - 推荐方式：**不要手动复制/编辑 `.env` 来"猜配置"**，直接走安装向导，让系统自动写入数据库与应用 URL

### 安装向导的 .env 写入规则（以当前代码实现为准）
 
 - 若 `.env` 不存在：会从 `.env.example` 复制生成 `.env`
 - 若 `.env` 已存在：会先备份为 `.env.backup.{timestamp}`
 - 会写入/更新：`APP_URL`、`DB_CONNECTION`、`DB_*`（SQLite 或 MySQL）
 - 若 `APP_KEY` 缺失或为空：会自动生成并写入

小提示：
 
 - **`APP_URL` 建议填写你实际访问的地址**（例如 `http://localhost:9445` 或线上域名），避免出现回调地址不一致等问题。
 - 生产环境建议将 `APP_ENV` 设置为 `production`，并关闭 `APP_DEBUG`（避免敏感信息泄露）。

## 安装完成后：启动与部署方式
 
### 本地一键启动（推荐）
 
 项目提供 `bin/start.sh` 和 `bin/start.bat` 用于并发启动：Laravel Server、队列、Vite。
 
 适合本地开发：你不需要分别开多个终端，一个命令就能把常用的开发进程都跑起来。
 
 ```bash
 # Linux/Mac
 ./bin/start.sh
 
 # Windows
 bin/start.bat
 ```
 
 如果你看到控制台中有 `server` / `queue` / `vite` 等进程的输出，说明启动正常。端口以控制台输出为准。

> 💡 **如何停止？**：在终端按 `Ctrl + C` 即可停止所有进程。

> 💡 **端口冲突怎么办？**：如果默认端口被占用，可以通过修改 `start.sh` / `start.bat` 中的端口配置，或者在启动命令中指定端口（例如 `php artisan serve --port=8081`）。

### 本地分开启动（可选）
 
 如果你更习惯“一个进程一个终端”，也可以分别启动后端与前端。
 
```bash
php artisan serve --host=0.0.0.0 --port=9445
```
 
 ```bash
 npm run dev
 ```

```bash
php artisan queue:work --sleep=3 --tries=1 --timeout=300
```


## 页面安装步骤（管理端安装向导）

当服务已经可访问后，在浏览器打开安装向导(`/install`)，按步骤填写信息即可完成初始化。  
如果你已经通过命令行完成安装，则无需再走页面安装。

> 🎯 **安装向导会做什么？**
>
> - Step 1：环境检查（PHP 版本、扩展、目录权限等）
> - Step 2：填写数据库配置和管理员账号
> - Step 3：确认信息并执行安装
>
> 整个过程大约需要 1 分钟，主要时间花在创建数据库表上。

### 数据库选择
 
 - SQLite：更适合本地/开发环境
 - MySQL：更适合生产环境
 
 MySQL 情况下，如果你填写的数据库用户有权限，安装流程会尝试创建数据库（若不存在）。建议为 Mosure 单独创建数据库与账号，并授予所需权限。

> 💡 **SQLite vs MySQL 怎么选？**
>
> - **SQLite**：文件型数据库，无需单独安装数据库服务，适合快速体验和本地开发。但并发性能较弱，不适合高并发生产环境。
> - **MySQL**：传统的关系型数据库，性能更好，支持高并发。需要单独安装 MySQL 服务，适合生产环境。
>
> 如果你只是想快速体验 Mosure，选 SQLite 就够了；如果准备长期使用或部署到生产环境，建议用 MySQL。

> 💡 **如果 SQLite 提示只读数据库**：通常是 `database/` 目录或 `mosure.db` 文件对当前 PHP 运行用户不可写。安装器会尽量自动创建并修正权限，但在受限服务器环境中，你仍可能需要手动将 `database/` 目录授权给 PHP/Web 用户。

### 安装完成后的标记
 
 - 安装完成后会写入 `.locked`（存在时视为已安装）
 
 如果你需要"重新安装"，请务必先确认是否会影响现有数据（例如数据库是否需要重置）。

> ⚠️ **警告**：重新安装会重置系统核心表，可能导致现有数据丢失。如果你只是想修改配置，建议直接编辑 `.env` 或通过管理界面修改，不要轻易重新安装。

> 💡 **如何临时禁用安装向导？**：手动创建 `.locked` 文件即可（例如 `echo "locked" > .locked`）。安装向导检测到 `.locked` 存在时会跳过安装流程。

## 登录
 
 安装成功后，你可以直接前往登录页。
 
 - 登录入口：`/login`

> 💡 **首次登录提示**：使用你在安装向导中创建的管理员账号登录。登录后建议立即修改密码（如果需要），并检查系统配置。

## 生产部署要点

 以下内容为通用部署要点，具体参数请按你的环境调整。生产部署的目标是：
 
 - 代码与依赖稳定
 - 静态资源提前构建
 - 缓存/队列/权限配置到位
 - 运行进程可守护、可重启

> 🎯 **生产部署检查清单**
>
> - [ ] Web Server 配置正确（根目录指向 `public/`）
> - [ ] PHP-FPM 版本 >= 8.2
> - [ ] `.env` 配置正确（`APP_ENV=production`、`APP_DEBUG=false`）
> - [ ] 前端资源已构建（`npm run build`）
> - [ ] 目录权限正确（`storage/`、`bootstrap/cache/` 可写）
> - [ ] 队列进程已启动并配置守护
> - [ ] 数据库备份策略已配置(可选)
> - [ ] HTTPS 已启用(可选)
> - [ ] 日志监控已配置(可选)

### Web Server
 
 - Web 根目录指向：`public/`
 - PHP-FPM 版本建议与项目要求一致（>= 8.2）

 可选用 nginx、apache 或其他支持 PHP 的 Web 服务器。它们只在生产部署时需要，本地一键安装和开发运行不依赖 Nginx。

### 静态资源构建
 
 生产环境建议在部署阶段构建一次前端静态资源。
 
 ```bash
 npm run build
 ```

> 💡 **为什么生产环境要提前构建？**：开发模式下，Vite 会实时编译前端资源，适合开发调试。但生产环境应该使用预构建的静态文件，以提高性能和稳定性。

> ⚠️ **构建失败怎么办？**：如果构建失败，检查 Node.js 版本是否符合要求，并尝试删除 `node_modules/` 后重新安装依赖（`rm -rf node_modules && npm install`）。

### 队列与异步任务
 
 - 项目在开发模式下通过 `php artisan queue:work ...` 启动队列
 - 生产环境建议使用进程管理工具托管队列进程

> 💡 **为什么需要队列？**：Mosure 的某些操作（如导出、AI 处理等）是异步执行的，通过队列来处理。如果队列进程没有运行，这些操作会一直处于"处理中"状态。

> 💡 **推荐工具**：可以使用 Supervisor 来托管队列进程，确保进程崩溃后自动重启。配置示例可以参考 Laravel 官方文档。

### 目录权限

 - 需要可写：`storage/`、`bootstrap/cache/`
 - 安装器会自动创建以下目录：`storage/framework/cache`、`storage/framework/sessions`、`storage/framework/views`、`storage/logs`、`bootstrap/cache`
 - 安装流程会尝试创建 `public/storage` 符号链接（等价于 `php artisan storage:link`）
 
 如果线上没有符号链接权限，你可以在部署时手动处理该链接（或调整 Web Server 的静态资源映射策略）。

> 💡 **如何设置权限？**：通常设置为 `755` 或 `775`，所有者为 Web Server 运行用户（例如 `www-data` 或 `nginx`）。例如：
>
> ```bash
> chmod -R 775 storage bootstrap/cache
> chown -R www-data:www-data storage bootstrap/cache
> ```

> ⚠️ **权限问题常见症状**：上传文件失败、日志无法写入、缓存无法更新等。

> 💡 **即使安装器已自动建目录，受限服务器仍可能需要手动授权**：尤其是宝塔、面板托管或容器挂载目录场景，若 PHP CLI 用户和 Web 用户不同，仍建议执行一次 `chown/chmod` 校正目录权限。

### 常用清理命令
 
 当你调整 `.env` 或部署了新代码后，遇到配置未生效、页面报错等情况，可优先尝试清理缓存。
 
 ```bash
 php artisan cache:clear
 php artisan config:clear
 ```

> 💡 **为什么需要清理缓存？**：Laravel 会缓存配置和路由以提高性能。当你修改 `.env` 或配置文件后，需要清理缓存才能让修改生效。

> 💡 **其他有用的清理命令**：
>
> ```bash
> php artisan view:clear      # 清理视图缓存
> php artisan route:clear     # 清理路由缓存
> php artisan config:cache    # 重新缓存配置（生产环境推荐）
> ```

## Docker（可选）

Docker 也属于“先部署服务，再进行页面安装”的流程；这一部分整体逻辑无需改动。

项目提供了完整的 Docker 部署方案，包含以下文件：

- `Dockerfile` - 多阶段构建配置
- `docker-compose.yml` - 服务编排配置
- `.env.docker.example` - Docker 环境变量模板
- `docker/` - 辅助配置文件

### 前置要求

- Docker Engine 20.10+
- Docker Compose 2.0+

### 快速开始

1. **克隆项目并进入目录**

```bash
git clone https://gitee.com/gibson_0822/mosure.git
cd mosure
```

2. **配置环境变量**

```bash
# 复制 Docker 环境变量模板
cp .env.docker.example .env.docker
```

然后编辑 `.env.docker`：

- `APP_URL`：修改为当前机器的外网 IP 或域名
- `APP_KEY`：重新生成

默认开放 `9445` 端口，如需修改可以在 `docker-compose.yml` 中调整：

```yaml
ports:
  - "9445:80"
```

编辑配置文件示例：

```bash
vim .env.docker
```

主要配置项说明：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `DB_DATABASE` | mosure | 数据库名称 |
| `DB_USERNAME` | mosure | 数据库用户名 |
| `DB_PASSWORD` | secret | 数据库密码 |
| `DB_ROOT_PASSWORD` | root | MySQL root 密码 |

> ⚠️ **安全提示**：生产环境请务必修改默认密码！

3. **构建并启动服务**

```bash
docker compose up -d
```

首次启动会自动执行以下操作：
- 构建应用镜像（包括 PHP 依赖和前端资源）
- 启动数据库、应用、队列和 Web 服务

4. **访问应用**

- 应用地址：http://localhost:9445
- 默认账号：见安装向导或首次运行提示

### 服务说明

Docker Compose 会启动哪些服务，以仓库中的 `docker-compose.yml` 为准。下面是一个典型示例：

| 服务名 | 容器名 | 说明 | 端口 |
|--------|--------|------|------|
| app | mosure_app | PHP-FPM 应用服务 | 9000 |
| queue | mosure_queue | Laravel 队列处理 | - |
| web | mosure_web | Web 服务入口 | 9445:80 |
| mysql | mosure_mysql | MySQL 数据库 | 3306 |

### 常用命令

```bash
# 查看服务状态
docker compose ps

# 查看服务日志
docker compose logs -f

# 查看特定服务日志
docker compose logs -f app
docker compose logs -f web

# 停止所有服务
docker compose stop

# 启动所有服务
docker compose start

# 重启所有服务
docker compose restart

# 停止并删除所有服务（保留数据卷）
docker compose down

# 停止并删除所有服务（包括数据卷）
docker compose down -v

# 重新构建并启动
docker compose up -d --build

# 进入应用容器
docker compose exec app bash

# 运行 Artisan 命令
docker compose exec app php artisan migrate
docker compose exec app php artisan cache:clear
docker compose exec app php artisan queue:work
```

### 数据持久化

以下数据通过 Docker Volume 持久化：

- `mysql_data` - MySQL 数据
- `storage` - Laravel storage 目录
- `bootstrap_cache` - Laravel 缓存

即使删除容器，数据也不会丢失。如需完全清理数据：

```bash
docker compose down -v
```

### 自定义配置

#### 修改端口

编辑 `docker-compose.yml`，修改 web 服务的端口映射：

```yaml
web:
  ports:
    - "8081:80"  # 将 9445 改为 8081
```

#### 修改 PHP 配置

创建 `docker/php/php.ini` 文件，Dockerfile 会自动加载。

#### 修改 Web 服务配置

如果你的 Docker Web 服务使用 Nginx，可编辑 `docker/nginx/default.conf`，然后重新构建：

```bash
docker compose up -d --build
```

### 故障排查

#### 1. 容器无法启动

```bash
# 查看容器状态
docker compose ps

# 查看容器日志
docker compose logs <service-name>
```

#### 2. 数据库连接失败

确认 `.env.docker` 中的数据库配置与 `docker-compose.yml` 中的 MySQL 服务配置一致。

#### 3. 权限问题

```bash
# 修复 storage 目录权限
docker compose exec app chown -R www-data:www-data storage bootstrap/cache
```

#### 4. 队列不工作

```bash
# 手动启动队列处理
docker compose exec app php artisan queue:work --sleep=3 --tries=3
```

### 生产环境部署

生产环境建议进行以下优化：

1. **修改默认密码** - 更新 `.env.docker` 中的数据库密码
2. **配置 HTTPS** - 使用反向代理或网关配置 SSL 证书
3. **设置资源限制** - 在 `docker-compose.yml` 中添加资源限制
4. **配置日志轮转** - 避免日志文件过大
5. **定期备份** - 备份 MySQL 数据和 storage 目录

> ⚠️ **注意**：Docker 部署需要一定的 Docker 知识，如果你不熟悉 Docker，建议先熟悉基本概念后再尝试。

## 常见问题
 
 ### 1) 安装向导提示缺少 PHP 扩展
 
 - 处理思路：安装缺失扩展后重启 PHP（FPM/Apache），然后刷新安装页面重新检测。

> 💡 **如何安装 PHP 扩展？**：不同系统的安装方式不同。
>
> - **Ubuntu/Debian**：`apt-get install php-{extension-name}`
> - **CentOS/RHEL**：`yum install php-{extension-name}`
> - **macOS（Homebrew）**：`brew install php@8.2`（通常包含常用扩展）
>
> 安装后记得重启 PHP 服务（例如 `systemctl restart php-fpm` 或 `brew services restart php@8.2`）。

### 2) 安装提示找不到 `.env.example`
 
 - 处理思路：确认仓库根目录存在 `.env.example`，并检查文件权限（Web 进程需可读）。

> 💡 **如何检查文件权限？**：使用 `ls -la .env.example` 查看权限。确保 Web Server 运行用户有读取权限（至少 `644`）。

> ⚠️ **常见原因**：
> - 文件被误删或移动
> - 文件权限设置不当
> - Git 克隆时使用了 `--depth=1` 导致文件不完整（通常不会）

### 3) MySQL 连接失败
 
 - 处理思路：确认主机/端口可达、账号密码正确、数据库用户权限足够；必要时先用命令行或 GUI 工具测试连接。

> 💡 **如何测试 MySQL 连接？**
>
> ```bash
> mysql -h {host} -P {port} -u {username} -p
> ```
>
> 如果能成功连接，说明数据库配置没问题。如果失败，检查：
> - 主机地址是否正确（本地用 `localhost` 或 `127.0.0.1`）
> - 端口是否正确（默认 3306）
> - 用户名和密码是否正确
> - 数据库用户是否有足够权限
> - 防火墙是否阻止了连接

### 4) 安装后页面 500 / 配置不生效
 
 - 处理思路：检查 `storage/logs` 日志；并尝试执行缓存清理（见上面的常用清理命令）。

> 💡 **如何查看日志？**
>
> ```bash
> tail -f storage/logs/laravel.log
> ```

> 日志中通常会包含具体的错误信息，例如：
> - 类找不到（依赖未安装）
> - 数据库连接失败
> - 文件权限问题
> - 配置错误

> ⚠️ **常见原因**：
> - `.env` 配置错误（例如数据库连接信息不对）
> - 依赖未完全安装（`vendor/` 或 `node_modules/` 不完整）
> - 缓存未清理（修改配置后未清除缓存）
> - PHP 版本不符合要求
> - 目录权限问题

> 💡 **快速排查步骤**：
>
> 1. 检查 `.env` 配置是否正确
> 2. 执行缓存清理命令
> 3. 检查 `storage/logs/laravel.log` 日志
> 4. 确认 PHP 版本和扩展
> 5. 检查目录权限
