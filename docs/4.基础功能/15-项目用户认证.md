# 项目用户认证

## 什么是项目用户认证？

项目用户认证是 Mosure 为单个项目提供的前台用户登录能力。它和系统管理员账号不同：

- **系统管理员账号**：用于登录 Mosure 后台，管理项目、模型、API Key、插件等。
- **项目用户账号**：用于某个项目的外部前端应用，例如 H5、Web、App、小程序等。

启用后，外部前端可以通过项目用户登录态直接调用当前项目的 `/open/*` 接口，Mosure 会根据项目用户角色权限进行校验。

## 适用场景

项目用户认证适合这些场景：

- 会员中心、用户投稿、前台内容发布等需要普通用户登录的项目；
- 没有独立 BFF/业务后端，但希望前端安全调用部分写接口；
- H5、App、小程序等多端共享同一套项目用户登录机制。

如果项目只是公开展示内容，通常只需要只读 API Key，不必启用项目用户认证。

## 启用或关闭项目用户认证

1. 进入目标项目。
2. 打开左侧菜单：**项目管理 → 项目配置**。
3. 进入 **认证/前台用户** 分类。
4. 打开或关闭 **启用项目用户认证**。
5. 点击 **保存**。

开启后，可以继续配置：

- **认证方式**：当前支持本地账号。
- **允许公开注册**：开启后外部前端可调用注册接口。
- **要求邮箱验证**：预留策略，启用前应先配置邮件服务。
- **登录态有效期（分钟）**：控制项目用户登录态 token 的过期时间。
- **允许的前端 Origin**：用于项目用户登录和 OpenAPI 直连的前端来源辅助校验。

## 项目用户与角色管理

项目用户认证开启后，左侧菜单会显示一级菜单：**项目用户**。

该页面包含两个页签：

- **项目用户**：创建、编辑、删除项目用户，并分配角色。
- **项目角色**：创建、编辑、删除自定义角色，并配置角色权限。

系统会初始化默认角色，例如：

- `member`：普通用户，默认只读。
- `editor`：编辑用户，可进行部分内容写入。

角色权限会决定项目用户登录态可以访问哪些 `/open/*` 接口。

## 对外认证接口

项目用户认证接口定义在 OpenAPI 路由中：

```http
POST /open/auth/{projectPrefix}/login
POST /open/auth/{projectPrefix}/register
GET  /open/auth/{projectPrefix}/me
POST /open/auth/{projectPrefix}/logout
```

例如项目 `kxm11`：

```http
POST /open/auth/kxm11/login
POST /open/auth/kxm11/register
GET  /open/auth/kxm11/me
POST /open/auth/kxm11/logout
```

### 登录

```http
POST /open/auth/{projectPrefix}/login
Content-Type: application/json
```

请求体：

```json
{
  "account": "user@example.com",
  "password": "123456"
}
```

`account` 可以是邮箱、用户名或手机号。

成功后返回项目用户登录态：

```json
{
  "code": 200,
  "message": "login_success",
  "data": {
    "token": "pu_kxm11_xxx",
    "expires_at": "2026-04-20 12:00:00",
    "user": {}
  }
}
```

### 注册

```http
POST /open/auth/{projectPrefix}/register
Content-Type: application/json
```

请求体：

```json
{
  "email": "user@example.com",
  "username": "user001",
  "name": "张三",
  "password": "123456"
}
```

说明：

- `email` 和 `username` 至少填写一项。
- 注册接口只有在 **允许公开注册** 开启后可用。
- 注册成功后同样返回 `pu_*` 登录态。

### 获取当前项目用户

```http
GET /open/auth/{projectPrefix}/me
Authorization: Bearer pu_kxm11_xxx
```

### 退出登录

```http
POST /open/auth/{projectPrefix}/logout
Authorization: Bearer pu_kxm11_xxx
```

退出后，当前登录态会被撤销。

## 调用 OpenAPI

项目用户登录后，前端调用其它 `/open/*` 接口时携带登录态：

```http
Authorization: Bearer pu_kxm11_xxx
```

例如：

```bash
curl -X POST 'https://example.com/open/content/create/article' \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer pu_kxm11_xxx' \
  -d '{"title":"示例文章"}'
```

Mosure 会根据项目用户的角色权限校验当前接口是否允许访问。

## 与 API Key 的区别

| 方式 | 适用场景 | 是否建议放前端 |
| --- | --- | --- |
| 只读 API Key | 公开内容展示 | 可以 |
| 自定义 API Key | 少量低风险写入或服务端集成 | 谨慎 |
| 全部权限 API Key | BFF/服务端集成 | 不可以 |
| 项目用户登录态 `pu_*` | H5、Web、App、小程序等项目用户登录 | 可以 |

项目用户登录态用于识别普通项目用户，并按角色权限限制 `/open/*` 调用；API Key 更适合服务端系统集成。

## 多前端支持

项目用户登录态不区分 `client_type`。H5、Web、App、小程序等前端都使用同一套 Bearer token 机制：

```http
Authorization: Bearer pu_{projectPrefix}_{random}
```

不同前端只需要自行选择合适的保存方式，例如：

- Web/H5：本地存储或 Cookie（按业务安全要求选择）。
- App/小程序：安全存储或平台提供的本地存储能力。

## 安全建议

- 不要把系统管理员账号当作项目用户使用。
- 不要把全权限 API Key 放入浏览器前端、App 包或公开仓库。
- 项目用户角色应按最小权限配置。
- 对公开注册场景，应配合验证码、频率限制、审核机制等策略。
- 如果已有独立业务后端，仍可使用 BFF 方式：前端请求业务后端，业务后端持有 API Key 调用 Mosure。
