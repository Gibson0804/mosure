# 产品定位与场景设想

本文档用于记录 Mosure 当前阶段的产品目标、定位思考、目标用户、首页文案方向与对外介绍版本，作为后续产品演进、文档编写与功能取舍的参考。

---

## 一句话定位

Mosure 希望成为一套面向个人与小团队的：

**可私有部署的结构化知识与内容管理中台。**

它不仅用于保存知识，还要能把知识结构化为内容模型，并通过 OpenAPI、页面、插件等方式输出为实际可用的应用能力。

---

## 一、产品定位梳理

### 1. 产品本质

Mosure 不是单纯的博客系统，也不是传统意义上的企业 CMS。

它更接近于：

- 一个可私有部署的知识与内容中台
- 一个支持结构化建模的内容管理系统
- 一个能够把内容继续输出为接口和应用的数据底座

### 2. 核心定位关键词

当前最适合概括 Mosure 的关键词是：

- 私有部署
- 知识沉淀
- 结构化内容
- OpenAPI 输出
- 面向应用连接

### 3. 与传统 CMS 的区别

传统 CMS 更强调：

- 页面发布
- 栏目管理
- 文章管理

而 Mosure 更强调：

- 知识与内容统一管理
- 抽象为内容模型
- 一份内容，多处复用
- 面向 API、页面、AI 应用、客户端输出

### 4. 与通用知识库工具的区别

通用知识库更偏：

- 写文档
- 做协作
- 做组织沉淀

Mosure 则更进一步：

- 不只记录知识
- 还可以将知识结构化
- 并把结构化内容提供给应用使用

### 5. 当前阶段最清晰的定位表达

现阶段建议把 Mosure 定位为：

**面向个人与小团队的可私有部署结构化知识与内容中台。**

## 二、目标用户画像

### 用户画像 A：独立开发者 / 技术型个人

#### 典型特征

- 有自己的知识积累
- 有搭建个人产品或小工具的能力
- 希望知识、文档、接口能统一在一套系统里

#### 主要需求

- 管理技术文档、知识条目、教程、资源
- 定义内容模型
- 对外输出 API 给网站、App、AI 工具使用
- 希望可私有部署，数据掌握在自己手中

#### 他们为什么可能使用 Mosure

- 比纯文档工具更结构化
- 比传统 CMS 更适合知识型内容
- 比从零开发一个内容后端更省时间

### 用户画像 B：小公司创始人 / 工作室负责人

#### 典型特征

- 团队规模小
- 预算有限
- 不希望同时维护多个系统

#### 主要需求

- 管理企业知识、产品资料、FAQ、案例、文章
- 同时支持官网内容、帮助中心和内部知识库
- 通过接口给 H5、小程序、内部系统供数

#### 他们为什么可能使用 Mosure

- 一套系统解决多种内容管理问题
- 可逐步扩展，不必一开始上大型系统
- 适合私有部署与定制

### 用户画像 C：内容运营 / 知识管理负责人

#### 典型特征

- 不一定懂开发
- 更关注内容组织与复用效率
- 希望内容能标准化、结构化

#### 主要需求

- 建立知识分类和内容模型
- 管理内容版本、状态、字段和关系
- 将内容交给多个前端或业务系统使用

#### 他们为什么可能使用 Mosure

- 内容建模能力明确
- 不是单纯“写文章”
- 可以逐步形成内容中台

### 用户画像 D：AI 应用构建者

#### 典型特征

- 想为 AI 助手、Agent、问答系统准备高质量数据源
- 关注知识结构化和接口可用性

#### 主要需求

- 将知识沉淀为结构化内容
- 为 AI 应用提供可控、可查询、可维护的数据源
- 希望接口清晰且方便集成

#### 他们为什么可能使用 Mosure

- Mosure 天然适合作为知识和内容底座
- 比零散文档更容易被 AI 应用消费

## 三、当前期望解决的问题

很多个人和小公司在内容与知识管理上会同时遇到这些问题：

- 知识分散在文档、表格、聊天记录和零散系统中
- 内容可以写出来，但不够结构化，难以复用
- 传统 CMS 适合发内容，但不适合做知识沉淀
- 通用知识库适合写文档，但不方便输出成 API 和应用
- 想做一个专属小应用时，往往需要额外开发

Mosure 希望把这些能力连接起来：

- 先沉淀知识
- 再结构化建模
- 再输出为内容接口
- 最终服务于网站、知识库、AI 应用或内部工具

---

## 四、核心能力理解

Mosure 当前最重要的能力组合不是单一 CMS，而是下面四个层次：

### 1. 知识沉淀

- 保存个人或公司的长期知识
- 适合文档、知识条目、经验库、FAQ、规范等内容

### 2. 内容建模

- 将知识抽象为结构化内容模型
- 支持字段、关系、分类、版本等能力

### 3. 内容输出

- 自动生成 OpenAPI
- 支持页面、接口、前端应用、插件等消费方式

### 4. 应用连接

- 让同一份内容服务于多个具体应用
- 如官网、知识库、内部系统、AI 助手、客户端应用等

## 五、当前最适合落地的典型场景

### 场景一：企业或团队知识库

- 团队文档
- 流程规范
- 产品资料
- 售前售后 FAQ

### 场景二：结构化内容中心

- 文章、案例、产品资料、教程、术语库统一管理
- 一份内容，多处复用

### 场景三：帮助中心 / 文档中心

- 面向官网、客户支持、自助服务的内容输出

### 场景四：AI 应用的数据底座

- 将知识组织为可被 AI 使用的结构化内容
- 作为 Agent、聊天助手、检索系统、工作流系统的数据来源

### 场景五：小型业务应用的内容后端

- 通过 OpenAPI 给小程序、H5、内部工具、轻应用提供内容能力

## 六、产品边界

为了避免项目失焦，Mosure 当前不应把自己描述成“大而全的一切平台”，而应更明确边界：

### Mosure 更像什么

- 知识与内容中台
- 结构化内容管理系统
- 面向应用输出的内容后端

### Mosure 暂时不强调什么

- 全能低代码平台
- 复杂 ERP / OA / CRM
- 面向超大组织的完整数字化平台

---

## 七、市场判断

当前判断是：

### 1. 有市场

尤其是在个人、工作室、小团队、内容型小公司这一层。

### 2. 属于细分市场

它不是“大众级通用 CMS”竞争，而是“私有知识 + 结构化内容 + API 输出”这一细分方向。

### 3. 差异化机会在于组合能力

如果只是做 CMS，竞争会很激烈；  
如果能把下面三点组合清楚，就更容易形成差异化：

- 私有部署
- 结构化建模
- OpenAPI / 应用输出

## 九、README 产品介绍版

以下内容适合作为 README 顶部的产品介绍。

### README 短版

Mosure 是一套面向个人与小团队的可私有部署结构化知识与内容管理中台。它支持将知识沉淀为内容模型，并通过 OpenAPI、页面、插件等方式输出到网站、客户端、内部系统和 AI 应用中。

### README 中版

Mosure（Headless CMS）是一套可私有部署的知识与内容中台，适合个人、工作室和小公司使用。  
它不仅能管理文章、文档和媒体资源，还强调将知识抽象为结构化内容模型，并自动生成 OpenAPI，让同一份内容可以服务于官网、知识库、帮助中心、H5、小程序以及 AI 应用等多种场景。

### README 卖点版

Mosure 关注的不是“再做一个 CMS”，而是帮助用户完成下面这件事：

> 把分散的知识沉淀下来，变成结构化内容，再连接成真正可用的应用能力。

它适合用来搭建：

- 企业或个人知识库
- 帮助中心 / 文档中心
- 内容中台
- 面向 AI 的知识数据底座
- 为外部应用提供内容接口的后端系统

## 十二、当前产品策略建议

### 1. 聚焦“小而实用”

优先服务个人与小团队，不急于追求大而全。

### 2. 聚焦“知识结构化”

不是只会写文章，而是让知识能建模、复用、连接应用。

### 3. 聚焦“输出能力”

让内容不只停留在后台，而能进入：

- OpenAPI
- 网站
- AI 应用
- 客户端
- 内部工具

### 4. 聚焦“私有可控”

强调用户对数据、部署、接口能力的掌控感。
